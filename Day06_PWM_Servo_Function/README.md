ccr = SERVO_MIN_CCR + (u32)angle * (SERVO_MAX_CCR - SERVO_MIN_CCR) / SERVO_MAX_ANGLE;

③ 实现了按键控制舵机角度变化：

按下 KEY1，舵机角度减小 15°
按下 KEY2，舵机角度增大 15°

实验现象
按一下 KEY1，舵机向左转小 15°；
按一下 KEY2，舵机向右转大 15°。
最大收获

今天最大的收获是从“直接设置 CCR”进步到了“输入角度控制舵机”。

以前是：

TIM_SetCompare1(TIM1, 500);
TIM_SetCompare1(TIM1, 2500);

现在是：

Servo_SetAngle(90);
Servo_SetAngle(135);
Servo_SetAngle(180);

这样代码更接近真实项目，也更方便以后接入智能车的转向控制算法。

最大 BUG
BUG004：长 delay 和重复设置角度导致按键控制迟钝

现象：

按键有时候有反应，有时候没反应，舵机反应迟钝。

原因：

while(1) 中每次循环都会执行：

delay_ms(1000);
Servo_SetAngle(135);

这导致两个问题：

1. delay 太长，按键扫描频率下降；
2. 每次循环最后都会强制设置舵机到 135°，覆盖了按键刚刚设置的角度。

解决：

删除 Servo_SetAngle(135)，并把：

delay_ms(1000);

改成：

delay_ms(20);

让主循环更频繁地扫描按键。


今天最困惑的是：

如果主循环里有比较长的 delay，为什么按键会变得不灵敏？
如果生活中很多按钮本来就是长按 1 秒或 3 秒才触发，那它和我今天遇到的按键反应迟钝是不是同一个原理？

后来理解到：

我今天遇到的问题是因为 delay_ms(1000) 让主循环扫描按键的频率下降了。程序每隔 1 秒才扫描一次按键，如果我短按 0.2 秒，很可能刚好发生在 delay 期间，程序就会错过这次按键。

而生活中“长按 1 秒 / 3 秒才触发”的按钮，虽然也和按键检测有关，但它不是因为程序扫描慢，而是程序故意设计成：必须连续检测到按键按下达到一定时间，才执行对应功能。

今天的新发现

今天发现，生活中很多长按按钮的底层逻辑，其实和 STM32 里的按键扫描、计数器、状态机思想很像。

比如长按 1 秒触发的逻辑可以理解为：

程序每隔 20ms 扫描一次按键
↓
如果检测到按键一直处于按下状态，就让计数器加 1
↓
当计数器累计到 50 次时，说明已经按下约 1 秒
↓
触发长按功能
↓
松手后计数器清零

也就是说：

1 秒 = 1000ms
1000ms / 20ms = 50 次

如果连续 50 次都检测到按键按下，就可以认为这是一次长按。

这个过程本质上就是：

按键扫描 + 计数器 + 状态机
对今天 BUG 的进一步理解

今天的 BUG 和生活中的长按按钮有相似点，但本质不同。

我的 BUG 是：

delay 太长，导致程序扫描按键太慢，所以容易错过短按。

生活中的长按按钮是：

程序扫描很快，但故意要求按键持续按够一定时间才触发。

所以以后写程序时，不能简单地用很长的 delay 来实现等待。
如果既想等待一段时间，又想保持按键灵敏，应该使用：

1. 短 delay + 计数器
2. 定时器中断计时
3. 状态机
4. 外部中断检测按键

这让我理解到，嵌入式程序里很多“等待”不能直接阻塞 CPU，而应该用非阻塞的方式来处理。
