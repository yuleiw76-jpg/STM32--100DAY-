① 中断的基本概念

中断可以理解为：

CPU 正在执行 main 函数里的任务时，如果某个中断事件发生，CPU 会暂时停下当前任务，转去执行对应的中断服务函数。中断服务函数执行完后，再回到 main 函数刚才被打断的位置继续执行。

执行流程大概是：

main 函数正常执行
↓
中断事件发生
↓
CPU 进入中断服务函数
↓
执行中断服务函数中的代码
↓
退出中断
↓
回到 main 函数刚才停止的位置继续执行
② 定时器中断

定时器中断就是：

利用 STM32 内部的定时器，每隔设定好的时间自动触发一次中断。

比如可以设置成每 1ms、10ms、500ms 或 1s 触发一次。

这个时间主要由两个参数决定：

PSC：预分频系数
ARR：自动重装载值

定时器中断周期公式可以理解为：

定时器中断周期 = (PSC + 1) × (ARR + 1) ÷ 定时器时钟频率
③ 为什么 PSC 和 ARR 要减 1

STM32 定时器内部规定：

实际分频系数 = PSC + 1
实际计数次数 = ARR + 1

所以如果我们想要 7200 分频，就不能直接写 7200，而是要写：

psc = 7200 - 1;

因为实际分频是：

7199 + 1 = 7200

ARR 也是一样。

如果我们想让定时器计数 10000 次，就写：

arr = 10000 - 1;

因为计数器是从 0 开始数的：

0, 1, 2, ..., 9999

这样一共正好是 10000 次。

所以减 1 的目的不是编译器自动处理，而是为了符合定时器硬件的计数规则，同时也方便我们人类按照“实际分频多少、实际计数多少”来计算。

④ 中断服务函数 TIMx_IRQHandler

定时器中断触发后，会进入对应的中断服务函数。

例如：

void TIM3_IRQHandler(void)
{
    // 中断触发后执行的代码
}

这个函数的作用是：

当 TIM3 定时器中断发生时，CPU 自动跳转到这里执行代码。

执行完中断服务函数后，CPU 才会回到 main 函数继续执行。

⑤ 为什么中断服务函数里不建议写 delay

中断服务函数应该尽量短，最好快进快出。

如果在中断服务函数里写：

delay_ms(100);

或者比较长的延时，会导致 CPU 长时间停留在中断里面。

如果定时器每 1ms 触发一次，而中断服务函数执行时间太长，就可能出现：

当前中断还没执行完
↓
下一次中断又来了
↓
中断请求被挂起
↓
当前中断结束后马上又进入下一次中断
↓
main 函数几乎没有机会执行

所以中断服务函数里更适合写短小的代码，比如：

flag = 1;
count++;
PCout(13) = !PCout(13);

不适合写：

delay_ms();
while();
复杂大循环;
长时间等待;
⑥ STM32 标准库

STM32 标准库是 ST 官方提供的一套外设驱动代码。

它的作用是：

把复杂的寄存器操作封装成函数，方便我们配置和控制 GPIO、定时器、串口等外设。

例如：

GPIO_Init();
GPIO_SetBits();
GPIO_ResetBits();

TIM_TimeBaseInit();
TIM_Cmd();
TIM_ITConfig();

这些都是标准库函数。

标准库和寄存器的关系是：

标准库函数
↓
封装寄存器操作
↓
修改 STM32 外设寄存器
↓
控制真实硬件

所以标准库本质上还是在操作寄存器，只是它把底层复杂操作封装得更容易使用。

⑦ stm32f10x_gpio.h/.c 和 stm32f10x_tim.h/.c

stm32f10x_gpio.h/.c 主要负责 GPIO 外设相关内容，包括：

1. IO 口初始化
2. 设置输出高低电平
3. 读取输入电平

其中：

gpio.h：负责声明 GPIO 相关函数、结构体、宏定义
gpio.c：负责实现 GPIO 相关函数

stm32f10x_tim.h/.c 主要负责定时器 TIM 外设相关内容，包括：

1. 定时器初始化
2. 预分频设置
3. 自动重装载设置
4. 开启定时器
5. 开启定时器中断
6. 清除中断标志位

其中：

tim.h：负责声明 TIM 相关函数、结构体、宏定义
tim.c：负责实现 TIM 相关函数

最大收获

今天最大的收获是理解了 PSC 和 ARR 的作用。

我知道了：

PSC 用来决定定时器时钟被分频多少
ARR 用来决定计数器数多少次后产生一次更新中断
PSC 和 ARR 组合起来可以决定定时器中断的时间

并且理解了为什么设置时通常要写成：

psc = 实际分频系数 - 1;
arr = 实际计数次数 - 1;

因为 STM32 定时器硬件内部的规则是：

实际分频系数 = PSC + 1
实际计数次数 = ARR + 1
最大 BUG

今天没有遇到明显 BUG。

今天最困惑的是

今天最困惑的是为什么 STM32 定时器规定实际分频系数等于 PSC + 1，实际计数次数等于 ARR + 1。后来理解到，这是因为硬件寄存器从 0 开始计数，PSC = 0 表示最小的 1 分频，ARR = 9999 表示从 0 数到 9999，一共 10000 次。这样设计不是多此一举，而是硬件计数方式决定的。
