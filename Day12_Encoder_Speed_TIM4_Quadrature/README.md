## 编码器为什么需要 A、B 两相

单独一相信号可以统计脉冲数量，却通常不能判断旋转方向。电机正转和反转时，单独观察 A 相都可能得到相似的方波。

A、B 两相存在固定的先后关系：

- A 发生边沿时观察 B 的电平。
- B 发生边沿时观察 A 的电平。
- 两相谁领先、谁滞后，可以区分两个相反的旋转方向。

“正转”和“反转”由 A/B 接线、观察方向和程序定义共同决定，并不是编码器天然固定的名称。

## 90° 相位差

A、B 相差 90°，主要指两路周期电信号相差四分之一个信号周期，也就是 90° 电角度。它不能简单理解为机械孔盘在空间中错开了 90°。

## A/B 状态变化

A、B 组合只有四种状态：

```text
00、01、10、11
```

一个方向可能按照下面的顺序变化：

```text
00 -> 10 -> 11 -> 01 -> 00
```

相反方向则可能是：

```text
00 -> 01 -> 11 -> 10 -> 00
```

两组状态顺序相反，因此可以用来判断方向。

## 每圈 1560 个计数

本课程按下面的参数估算输出轴一圈的计数：

```text
13 * 30 * 4 = 1560 counts/revolution
```

| 参数 | 含义 |
|---:|---|
| 13 | 厂家标称的基础脉冲或周期数，具体口径需查规格书 |
| 30 | 减速比，内部电机轴约转 30 圈时输出轴转 1 圈 |
| 4 | 一个基础正交周期包含 4 次有效状态跳变 |
| 1560 | 输出轴转 1 圈对应的理论 TIM4 计数 |

1560 的单位是 `count/revolution`，不是“1560 种状态”。A/B 组合状态始终只有四种。

厂家对 PPR、CPR 和边沿倍频的口径可能不同。转接端子到货后，必须手动转动输出轴一圈或多圈，记录真实计数并进行校准。

## 三个定时器的分工

| 定时器 | 作用 |
|---|---|
| TIM1 | 输出 PWM，通过 L298N 控制电机 |
| TIM4 | 接收 A/B 两相，执行正交解码并统计位置 |
| TIM3 | 每 0.1 s 读取位置变化并计算速度 |

一句话区分：TIM4 负责“转了多少、往哪个方向转”，TIM3 负责“隔多久计算一次速度”。

## TIM1 PWM 配置

主程序调用：

```c
TIM1_PWM_Init(3000 - 1, 0);
```

| 参数 | 实际值 |
|---|---:|
| TIM1 时钟 | 72 MHz |
| PSC | 0 |
| 实际分频系数 | 1 |
| ARR | 2999 |
| 实际计数次数 | 3000 |

PWM 频率：

```text
f_PWM = 72 MHz / ((PSC + 1) * (ARR + 1))
      = 72000000 / (1 * 3000)
      = 24 kHz
```

实际通道对应关系：

| STM32 引脚 | TIM1 通道 | 作用 |
|---|---|---|
| PA8 | TIM1_CH1 | 电机驱动的一路 PWM 控制 |
| PA9 | TIM1_CH2 | 电机驱动的另一路 PWM 控制 |

`main.c` 初始化完成后立即执行：

```c
TIM_SetCompare1(TIM1, 1500);
TIM_SetCompare2(TIM1, 0);
```

因此连接电源后，程序会直接尝试以约 50% 占空比向一个方向驱动电机，并没有等待按键确认。首次实物供电前必须架空车轮或断开机械负载，并准备立即断电。

## GPIO 与 TIM4 对应关系

`encode.c` 将 PB6、PB7 配置为浮空输入：

```c
GPIO_InitStructure.GPIO_Pin = GPIO_Pin_7 | GPIO_Pin_6;
GPIO_InitStructure.GPIO_Mode = GPIO_Mode_IN_FLOATING;
```

TIM4 默认映射为：

| STM32 引脚 | TIM4 通道 | 约定信号 |
|---|---|---|
| PB6 | TIM4_CH1 | 编码器 A 相 |
| PB7 | TIM4_CH2 | 编码器 B 相 |

源码注释只写了 TIM4_CH1、TIM4_CH2，没有直接把它们命名为 A、B。上表采用课程约定；交换 PB6、PB7 的 A/B 接线后，计数方向和 `speed` 符号通常会反转。

## TIM4 编码器模式

实际核心调用：

```c
TIM_EncoderInterfaceConfig(
    TIM4,
    TIM_EncoderMode_TI12,
    TIM_ICPolarity_BothEdge,
    TIM_ICPolarity_BothEdge
);
```

- `TIM_EncoderMode_TI12` 同时使用 TI1 和 TI2。
- 编码器 A/B 的有效跳变推动 CNT 增加或减少。
- TIM4 根据两相顺序自动判断方向。
- 普通定时器通常由内部时钟推动 CNT；编码器模式由外部 A/B 信号推动 CNT。

TIM4 的实际时基参数：

| 参数 | 实际值 |
|---|---:|
| PSC | 0 |
| ARR | 1559 (`1560 - 1`) |
| CNT 范围 | 0 到 1559 |
| 更新中断 | 已启用 |

代码虽然先写了 `TIM_CounterMode_Up`，但进入编码器模式后，计数方向由 A/B 两相状态决定，TIM4 仍可向上或向下计数。“Up”不能理解成编码器模式只允许向上计数。

## DIR 方向位

中断中使用：

```c
(TIM4->CR1 >> 4) & 0x01
```

它先读取 TIM4 的 CR1，再取出 DIR 位：

| DIR | 当前方向 | 代码行为 |
|---:|---|---|
| 0 | 向上计数 | `circle_count++` |
| 1 | 向下计数 | `circle_count--` |

DIR 是当前计数方向位，不是单独的“溢出方向寄存器”。

## 上下回绕和 `circle_count`

由于 ARR 为 1559：

```text
向上回绕：1558 -> 1559 -> 0
向下回绕：1 -> 0 -> 1559
```

向上回绕时：

```c
circle_count++;
```

向下回绕时：

```c
circle_count--;
```

更新中断结束前执行：

```c
TIM_ClearITPendingBit(TIM4, TIM_IT_Update);
```

`circle_count` 只记录跨过了多少个完整的 1560 计数周期，不是每个编码器脉冲都加减。普通有效边沿只会改变 `TIM4->CNT`。

累计位置的理论表示为：

```text
Position = circle_count * 1560 + TIM4->CNT
```

## TIM3 测速周期

主程序调用：

```c
TIM3_Init(10000 - 1, 720 - 1);
```

实际参数：

| 参数 | 实际值 |
|---|---:|
| TIM3 时钟 | 72 MHz |
| PSC | 719 |
| 实际分频系数 | 720 |
| 计数频率 | 100 kHz |
| 单次计数时间 | 10 us |
| ARR | 9999 |
| 实际计数次数 | 10000 |
| 更新中断周期 | 0.1 s |

计算过程：

```text
72 MHz / 720 = 100 kHz
10000 * 10 us = 0.1 s
```

## 实际速度计算

`tim.c` 中的实际代码为：

```c
speed = ((float)(
    (circle_count - last_count) * 1560
    + ((int)(TIM4->CNT) - last_num)
)) / 1560.0 / 0.1;

last_count = circle_count;
last_num = TIM4->CNT;
```

逐项解释：

- `circle_count - last_count`：完整计数周期变化量。
- 乘以 1560：转换成编码器计数。
- `TIM4->CNT - last_num`：当前周期内部计数变化量。
- 整个分子：两次采样之间的总计数变化量。
- 除以 1560：计数转换成输出轴圈数。
- 除以 0.1：0.1 s 内的圈数转换成每秒转数。

最终单位是 `r/s`。

每次计算后更新 `last_count` 和 `last_num`，这样下一次测速计算的是新的 0.1 s 区间，而不是一直减去最初位置。

## 为什么应保存 `current_num`

当前源码没有保存局部快照，而是直接读取了两次 `TIM4->CNT`：一次参与速度计算，一次写入 `last_num`。

硬件计数器会持续变化，两次读数可能不同。更一致的思路是先读取一次：

```c
int current_num = (int)TIM4->CNT;
```

本轮公式和 `last_num` 更新都使用同一个 `current_num`。这只是代码审查建议，本上传版本没有修改原始源码。

## `volatile` 检查

源码实际定义为：

```c
int circle_count = 0;
float speed;
```

两者都没有 `volatile`：

- `circle_count` 在 TIM4 中断中修改，在 TIM3 中断中读取。
- `speed` 在 TIM3 中断中修改，在主循环中读取。
- `last_count`、`last_num` 只在 TIM3 中断内部使用，通常不需要 `volatile`。

建议关注的声明形式为：

```c
volatile int circle_count;
volatile float speed;
```

`volatile` 可以要求编译器每次从内存重新读写，避免长期使用缓存旧值，但不能保证原子性、多步操作不可被打断、多变量快照一致或自动实现中断安全。

## OLED 显示

主循环每 10 ms 执行：

```c
sprintf(text, "%.2fr/s    ", speed);
OLED_ShowString(8, 2, (u8 *)text, 16);
```

显示格式特点：

- 保留两位小数。
- 单位为 `r/s`。
- 负值会由 `%f` 自动带 `-` 号。
- 正值不会强制显示 `+` 号。
- `char text[20]` 对正常电机转速字符串通常足够，但 `sprintf` 本身没有长度限制。

TIM3 每 100 ms 更新一次 `speed`，主循环却约每 10 ms 重写一次 OLED，刷新频率约高十倍，存在不必要的软件 I2C 开销。当前只记录，不修改。

## 代码中的潜在问题

### 1. 编码器配置后又重新初始化通道 1

代码先将两路极性都设为 `BothEdge`：

```c
TIM_EncoderInterfaceConfig(
    TIM4,
    TIM_EncoderMode_TI12,
    TIM_ICPolarity_BothEdge,
    TIM_ICPolarity_BothEdge
);
```

随后又只配置 TIM4 通道 1：

```c
TIM_ICInitStructure.TIM_Channel = TIM_Channel_1;
TIM_ICInitStructure.TIM_ICPolarity = TIM_ICPolarity_Rising;
TIM_ICInitStructure.TIM_ICSelection = TIM_ICSelection_DirectTI;
TIM_ICInitStructure.TIM_ICPrescaler = TIM_ICPSC_DIV1;
TIM_ICInitStructure.TIM_ICFilter = 0x00;
TIM_ICInit(TIM4, &TIM_ICInitStructure);
```

这可能覆盖通道 1 前面的部分配置，使 CH1 变为 Rising，而 CH2 仍保留前面的配置，形成两路不一致。这里的 `TIM_ICInitTypeDef` 和额外 `TIM_ICInit()` 可能冗余，需要结合参考手册和实物计数验证；目前不自动删除或修改。

### 2. `circle_count` 与 CNT 的回绕边界竞争

测速时需要组合读取 `circle_count` 和 `TIM4->CNT`。如果 TIM4 恰好在两次读取之间发生回绕，可能读取到“旧圈数 + 新 CNT”或其他不一致组合，使某一次速度出现约一整圈的跳变。

只保存 `current_num` 能解决同一寄存器重复读取的问题，但不能单独保证 `circle_count` 与 CNT 是同一时刻的原子快照。后续可学习临界区、重复校验或更稳健的位置快照方法，本次不修改中断逻辑。

### 3. 共享变量缺少 `volatile`

`circle_count` 和 `speed` 都跨执行上下文共享，但当前没有 `volatile`，存在编译器使用旧值的风险。

### 4. 注释与实际对象不一致

```c
TIM_Cmd(TIM4, ENABLE);  //使能TIM1
```

代码实际使能的是 TIM4，注释误写成 TIM1。

`TIM_CounterMode_Up` 的注释也容易让人误以为编码器只能向上计数，实际方向由编码器接口和 DIR 位决定。

### 5. 源码编码兼容性

多个原始 `.c/.h` 文件不是有效 UTF-8，可能采用 GBK/ANSI 编码。在 Keil 或中文 Windows 中可能正常显示，但 GitHub 按 UTF-8 解析时，部分中文注释可能出现乱码。本次为保证源码哈希和原工程一致，没有转换编码。

### 6. 主循环遗留内容

`main.c` 包含未实际使用的 `#include "key.h"` 和局部变量 `u8 a;`。Day12 主流程没有调用 `KEY_Init()` 或 `KEY_Scan()`，因此精简上传版没有复制 KEY 模块。

## 今日容易混淆的地方

- 90° 是电信号相位差，不是简单的机械孔位角度。
- A 发生边沿时看 B，B 发生边沿时看 A。
- 正反转由接线、观察方向和程序定义。
- 1560 是每圈计数，不是状态数。
- A/B 组合状态只有四种。
- TIM4 的 ARR 不是测速时间。
- TIM3 负责采样时间，TIM4 负责编码器计数。
- `TIM4->CNT` 是当前周期内计数，不是圈数。
- `circle_count` 不是每个脉冲都加减。
- DIR=0 向上计数，DIR=1 向下计数。
- 向下回绕是 0 -> ARR，不是到 ARR 后清零。

## 今日完整数据流

```text
TIM1 PWM
-> L298N
-> 电机转动
-> 编码器输出 A/B
-> PB6/PB7
-> TIM4 正交解码
-> CNT 加减
-> circle_count 处理回绕
-> TIM3 每 0.1 s 读取位置差
-> 计数换算成圈数
-> 计算 r/s
-> OLED 显示
```
